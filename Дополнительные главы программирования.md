# Лекция Р1: Жизненный цикл ПО

Жизненный цикл ПО (ЖЦПО) - период от рождения программы до ее забвения.

*Этапы разработки программного продукта*:
1. Постановка задачи. Оценка осуществимости;
Конечный этап этого шага, это принятие решения: будем ли реализовывать проект или нет.
Чтобы оценить возможно ли справиться с задачей или нет необходимо пройти несколько шагов:
* сбор информации о задаче (как компания работает, с чем работает, с какими данными). Данный этап выполняется в виде интервью (устных или письменных), серии встреч, мозговой штурм и т.д.;
* формулировка условия задачи. Никакая задача не может быть решена абстрактно, всегда есть некие ограничения.;
* определение конечных целей решения задачи;
* определение формы выдачи результатов;
* описание данных;
* *дополнение*: время, сроки, стоимость;

Если решение принято, то составляется документ: Техническое задание - официальный документ.
Без технического задания, начинать работу бессмысленно.
Выполнение этого этапа тоже должно оплачиваться, так как этот этап занимает не один день.

2. Анализ и исследование задачи, модели:
	- разработка математической модели;
	- разработка структур данных;
	- архитектура информационной системы;

Модель - это попытка в виртуальной среде представить реально существующие объекты или явления. Всегда когда модель строиться, всегда происходит упрощение. Любая модель проще, чем реальность, так как все детали не нужны.
Конечный этап этого шага - это составлено понимание о будущей системе и предметной области с которой работаем, но нет главного ответа на вопрос: "Что делать?" и "Как делать?"

3. Разработка алгоритмов:
	* выбор методов и подходов;
	* проектирование алгоритмов. Скорее всего алгоритм будет не один, необходимо выбрать оптимальные алгоритмы;

4. Программирование (кодирование):
	* выбор стека технологий (языка программирования и т.п.);
	* уточнение способа организации данных;
	* запись алгоритмов на языке программирования;

На данном этапе, программист подучает всю техническую документацию и должен воспроизвести все в программном коде. Шаг в сторону от задуманного не приветствуется (инициатива не приветствуется).
Возможности творческой реализации для программиста заключаются в отражении (реализации) концепции (алгоритмов) описанной в технической документации с помощью различных технологий и различных вариантов решения поставленных задач.
Хорошую математическую модель, можно испортить плохой реализацией и наоборот улучшить сделать более эффективной и интересной.
Конечный этап данного шага - это окончание этапа программирования (достижение промежуточного результата)

5. Тестирование и отладка:
	* выбор тестов и методов тестирования;
	* синтаксическая отладка;
	* отладка семантики и логической структуры;
	* тестовые расчеты и анализ результатов тестирования; 

Задача данного шага - выявить логические ошибки, которые приводят к некорректной работе программы и указать, в чем состоят ошибки, чтобы программист исправил ошибки.
На данном этапе, используются различные методы тестирования: нагрузочное тестирование, тестирование пользовательских интерфейсов и т.д.

Важная задача - это оценить количество ошибок в том или ином проекте для планирования работы сотрудников и т.д.
Как оценить количество ошибок в проекте: нужно сделать специально ошибки в коде, отдать тестировщикам и посмотреть, сколько ошибок они найдут. Если нашли 50 ошибок и все 50 ошибок нами созданы - это означает, что настоящих ошибок мало или просто нет. А если тестировщики нашли 50 ошибок и из них часть ошибок сделана не специально, то это означает, что в программе много ошибок.

6. Анализ результатов решения задач и уточнение, в случае необходимости, математической модели и возможный возврат с повторным выполнением шагов 2-5.

Чем к более раннему этапу осуществляется возврат, тем больше этапов необходимо будет провести заново.

7. Сопровождение программы:
	- составление документации к решенной задаче, к математической модели, к алгоритму, к программе, к набору тестов, к использованию;
	- подготовка к распространению, развертыванию программы (Deployment). Если необходимо, то закупка оборудования (сервер и т.д.), создание базы пользователей, распределение ролей пользователей и т.д.;
	- поддержка пользователей;
	- совершенствование программы;


## Жизненный цикл программного продукта

Разработка сложных информационных систем (продуктов), достаточно сложный процесс в зависимости от многих причин:
1. В разработке информационной системы задействовано довольно большое количество сотрудников: аналитики, владельцы продукта, специалисты по безопасности, специалисты по анализу данных, тестировщики, служба технической поддержки, служба по внедрению, служба по обучению пользователей;
2. Информационный продукт непростой. Это не 50 строчек кода и даже не 1000 строчек кода, это намного больше, причем современные информационные системы - это системы многокомпонентные, и если каждый из компонентов работает, то не факт, что они будут работать вместе.
3. Удовлетворение нужд заказчика. Система может быть сделана замечательно, но может прийти заказчик и сказать, что ему что-то не нравится.

В связи с указанными причинами, сроки разработки зачастую довольно большие, но также все зависит от проекта и от возможностей самой компании.
Поэтому, и появляется понятие жизненного цикла ПО, так как продукт постоянно развивается.

**Модель жизненного цикла информационной системы (Модель ЖЦ ИС)** - это структура, определяющая последовательность процессов, действий и задач, выполняемых на протяжении ЖЦ ИС, а также взаимосвязи между ними.

## Виды моделей жизненного цикла информационных систем:

### 1. Водопадная (каскадная) модель (Waterfall)

![[Pasted image 20251114122808.png]]

Специфика модели:
- Разбиение разработки на этапы;
- Последовательная организация работ;
- Выпуск полного комплекта документов при переходе на следующий этап;

*Примечание*: Возвращение на предыдущий этап не предусматривается, циклы возможны внутри самих этапов.

*Недостатки модели*:
- существенная задержка получения результатов;
- необходимость возврата на предыдущие этапы;
- сложность распараллеливания работ по проекту;
- информационная перенасыщенность каждого этапа;
- сложность управления проектом;
- высокий уровень риска и ненадежности инвестиций (каждая следующая команда зависит от выполнения предыдущего этапа);

Водопадная модель подходит для относительно простых задач: шаблонные, типовые задачи.

### 2. Спиральная модель

![[Pasted image 20251114124815.png]]

*Специфика модели*:
- Итерационный процесс работ;
- Активное использование прототипов (промежуточная версия, которая может не включать некоторые компоненты);
- Система версий;
- Постепенная конкретизация;
- Постепенная интеграция компонентов в сложных системах. Необходимо учитывать, что разработку, тестирование и апробацию одного компонента можно делать без остальных компонентов. Отсутствие иных компонентов не должно мешать процессу;
- Снижение уровня рисков;
- Упрощение внесения тактических изменений;

*Недостатки модели*:
- Определение момента перехода на следующий этап. Сложность временной оценки;
- Бесконечное совершенствование (бесконечную спираль улучшений, когда улучшения следуют одно за другим, и это отодвигает сроки финальной сдачи);
- Согласование итогов работы, согласование исходного задания, взаимодействие между разными отделами (Различные отделы могут находится на противоположных концах спирали);

### 3. Agile модель

Agile можно сравнить с философией или даже можно назвать это религией в сфере IT. Действительно ли модель Agile так хороша, вопрос не однозначный.

Модель Agile хорошо работает в относительно небольших коллективах, с новыми проектами, инновационными проектами, с проектами исследовательскими. 
Идеальная ситуация, когда модель Agile можно применять - это всевозможные стартапы, потому, что как раз для стартапа характерен небольшой коллектив (до 30 человек), неоднозначность в способах решения, так как стартап ежедневно сталкивается со сложностями, которые приходятся решать.

В больших корпорациях (Яндексе, Сбербанке, Тинькофф), вполне может быть то же самое, когда приходит команда к руководителю, предлагает новый проект и если получает одобрение, то тогда пытается разработать и запустить свой продукт внутри большой корпорации.

	- Приветствие изменения требований даже в конце разработки (это может повысить конкурентоспособность конечного продукта);
	- Частая поставка рабочего программного обеспечения (каждый месяц или неделю или чаще);
	- Тесное ежедневное общение заказчика с разработчиками на протяжении всего проекта. Иногда общение ради общения превалирует, так как что может произойти за один день, если вчера обсуждалась задача;
	- Проектом занимаются мотивированные личности, которые обеспечены нужными условиями работы, поддержкой и доверием;
	- Рекомендованный метод передачи информации - личный разговор (лицом к лицу);
	- Работающее программное обеспечение - лучший измеритель прогресса;
	- Спонсоры, разработчики и пользователи должны иметь возможность поддерживать постоянный темп на неопределенный срок;
	- Постоянное внимание к улучшению технического мастерства и удобному дизайну;
	- Простота - искусство не делать лишней работы;
	- Лучшие технологические требования, дизайн и архитектура получаются у самоорганизованной команды;
	- Постоянная адаптация к изменяющимся обстоятельствам. Команда должна систематически анализировать возможные способы улучшения эффективности и, соответственно, корректировать стиль своей работы.

Если внимательно посмотреть на тезисы в манифесте Agile, то уже становится ясно, что они не такие однозначные.

Agile методология имеет как преимущества, так и недостатки. Agile подход - это скорее неоднозначные исследовательские проекты, которые имеют довольно высокие риски при создании, при разработке, где на стартовом этапе до конца непонятен план и алгоритмы, где имеется только общее представление о будущей архитектуре, но хотим уточнять это в процессе и где есть небольшая управляемая команда. Для описанных ситуаций Agile методология подходит идеально.

1. Концептуальный дизайн строится на базе понимания будущего продукта заказчиками. Понимание формулируется на уровне обычного, общедоступного языка, без технических подробностей, только на том уровне, на котором понимает заказчик. Концептуальный дизайн формулируется в терминах, понятных и заказчику и разработчику

В данном пункте необходимо прийти к общему пониманию по следующим пунктам:
- Что мы хотим?
- Как это будет работать на уровне описания процессов;
- На уровне описания ролей;
- На уровне описания use case diagram (т.е. последовательности действий и прохождения); 
- На уровне описания данных, с которыми система будет работать;
- Что мы ждем от системы?
- Как мы видим ее в идеальном формате?

После того, как разработчик и заказчик пришли к общему понимаю по вышеуказанным вопросам, то необходимо составить документ и , подписать его с двух сторон. Так как именно по нему будет необходимо  отчитываться в будущем.

2. Логический дизайн - это уже со всеми IT подробностями, но на уровне схем, без физического написания кода без физического создания каких-то таблиц баз данных и так далее. Это логика: на уровне алгоритмов, на уровне моделей, на уровне архитектуры, на уровне всей системы в целом.

3. Физический дизайн - это самый нижний уровень, до конкретного кода, до конкретных функций, классов.

## Единая система программной документации

Документация связывает различные отделы между собой.

***Виды документов***:
- Техническое задание (В документе говорится, что заказчик хочет получить);
- Программа и методика испытаний (В документе описывается, какой должна быть программа, как ее тестировать и какие показатели тестов закладываются. Т.е. быстродействие было не хуже, чем вот такое-то. Надежность должна быть не хуже, чем такая-то. Документ разрабатывается, когда программы еще нет, но он составляется после всей подготовительной работы. При разработке программы, можно сверяться с данным документом для определения корректности работы программы);
- Руководство системного программиста (Документ, который рассказывает, как систему разворачивать, настраивать. Документ создается, когда программа готова);
- Руководство пользователя (Документ создается, когда программа готова);
- Руководство программиста SDK (Software Developer Kit) и API (Application Program Interface) - это документация для внешних разработчиков, которая помогает понять и использовать чужие  наработки;

API (Application Program Interface) - перечень декларации функций или классов, или констант, или чего-то еще и их спецификации с кратким описанием, что они делают, похоже на справку.


Спецификации ПО (в схемах):

# Лекция: Поиск ошибок и инструменты отладки

При написании программы, так или иначе возникают ошибки и это нормально, когда ошибки возникают. Необходимо уметь ошибки фиксировать, находить их причину и исправлять их.

*Можно выделить три вида ошибок*:
1. Ошибки синтаксические. 
	Связаны с неправильным написанием операторов, переменной, функции и т.д. Без исправления синтаксических ошибок, программу просто не откомпилировать. На данный момент, в средах программирования такие ошибки подсвечиваются и чаще всего предлагаются варианты их исправления;
2. Ошибки времени выполнения или exception. 
	Данные ошибки не связаны с синтаксисом, программа нормально написана, нормально запускается, компилируется и начинает работать, но в процессе работы возникают какие-то форс-мажорные обстоятельства.
	Чаще всего такого рода ошибки называются еще runtime error, то есть ошибки времени выполнения, или exception исключительной ситуации.
	Обычно, данные ошибки возникают из-за внешней среды (некорректный ввод пользователем данных и т.д.).
3. Ошибки логические.
	Самый сложный вид ошибок, так как:
		 Во-первых, непонятно, есть логическая ошибка или нет, и разработчик так и не может определить наличие этой ошибки до тех пор, пока уже пользователи ему на это не укажут. Одна из причин, почему такое происходит, это не стабильное возникновение ошибки;
		 Во-вторых, никто не скажет, что ошибка имеет место быть. 
		 В-третьих, еще есть менее очевидные логические ошибки, к примеру, результат правильный, но, память не освобождается (утечка памяти).
	Для поиска и устранения логических ошибок, обычно делают систему тестов.

Среди средств обнаружения логических ошибок можно использовать инструменты самой среды программирования.
### Инструменты отладки

Первое, что часто использует программист, это пошаговое выполнение кода.
Пошаговое исполнение кода может быть выполнено с помощью кнопок отладки на панели инструментов.

На панели инструментов есть три кнопки отладки:
- Шаг с заходом (step into) - заходит в выполняемую функцию;
- Шаг с обходом (step over) - не заходит в выполняемую функцию;
- Шаг с выходом (step out) - все команды, которые ниже курсора выполнятся и случится выход из функции;

*Примечание*: Кнопки отладки можно также найти в меню: "Откладка" и еще можно использовать комбинации клавиш.

В процессе отладки бывает полезным это инспектирование переменных, то есть мониторинг чему они равны и как они меняются.

Провести инспектирование переменных можно только в режиме отладки программы.

Способы просмотра значений переменных при инспектировании:
1. Просто навести мышку на переменную;
2. Использование окна: "Локальные значения". В данном окне показываются все переменные, которые видны в текущем контексте и их изменения.
3. Использование окна: "Контрольные значения" (watch). В данном окне можно добавить какую угодно переменную и наблюдать за ней. Оно удобно тем, если необходимо наблюдать за какой-то конкретной переменной или переменными. В окно можно самому записать значения, правда не любые, а только: переменные, константы и знаки операции, логические выражения (res > 0).

*Примечание*: Через меню: "Откладка", можно открыть до 4-х окон: "Контрольные значения".

**Точка остановки (breakpoint)** - позволяет остановить программу в определенном месте.

Для того, чтобы поставить точку остановки в Visual Studio, необходимо нажать на серой области слева от   точку очень легко надо вот на этой серой области слева от основного окна (окно, где написан сам код).
Точку остановки можно поставить там где нет пустых строк, комментариев и есть операторы, также можно поставить несколько точек остановки.

У точки остановки есть параметры и можно поставить условия, когда она выполняется.

Иногда бывает удобно не использовать инструменты отладки, а использовать обычные средства которые тоже могут быть применимы в процессе отладки.

В инструментах отладки много возможностей, также можно смотреть память, регистры,  проверять стек, утечку памяти, и так далее.


# Лекция Р2: Платформа .Net (dot Net)

C# - относительно молодой язык программирования, появился как логическое развитие языка C++.

Раньше, когда среды разработки только зарождались, не факт, что написанный код в одной среде, успешно будет скомпилирован в другой. Для успешной компиляции требовалось строгое соблюдение стандартов языка программирования и сведение к минимуму использование особенностей той или иной среды разработки.

Основа C# - это C++, есть схожие моменты в синтаксисе и некоторых приемах, но в C# были реализованы многие моменты, за которые ругали C++, в частности один из главных моментов: типобезопасный язык и язык безопасный к использованию памяти.

Сейчас, ситуация уже обратная, C++ стал догоняющим языком программирования, так как новые особенности добавляются с существенной задержкой и во многом повторяют то, что уже реализовано в C#. К примеру, в C# можно динамически управлять  массивами (можно легко создать массив и в процессе работы изменить его размер), а в C++ этого достичь было не так просто, а возможность динамически управлять массивами появилась совсем недавно. Тоже касается и строк, до некоторых пор в C++ строка была одномерным массивом, объявляемым самостоятельно с типом: "char" и с данным массивом приходилось работать, в то время как в C# - это отдельный тип: "string" и в переменную типа: "string"  можно записать текст и объединить его с другим текстом, применить функцию строковую и т.д.
Неопытный программист работая со стоками даже может не подозревать, что строка - это одномерный массив, так как в C# ничего это не выдает, а в C++ - это сложнее.
Сейчас C# больше похож на Java, чем на своего родителя C++ в идеологическом аспекте.

C# является объектно-ориентированным языком, который относится к категории RAD (rapid application development - быстрая разработка приложений)

Революционность C# состоит не только в возможностях самого языка, а еще и во всей инфраструктуре, которая его окружает.

В C# - это окружение называют: ".Net".

.Net - это платформа (где-то называется Framework) куда входит набор библиотек и единообразный поход к некоторым техническим операциям (например, к операции компиляции кода).

Кроме того, к ".Net" (примечание: dot Net) относят те компоненты, которые туда входят. К примеру, сборщик мусора - является универсальным компонентов для всех приложений ".Net" и не важно на каком языке это написано.

Идеология: ".Net" является открытой и любой разработчик или компания могут придумать свой язык, которой этой идеологии будут удовлетворять. К примеру, в продуктах компании: AutoCad используются языки, которые основаны на идеологии: ".Net".

Платформа: ".Net" ни с C# ни с Visual Studio не связана, ее можно установить самостоятельно. Если на компьютере развернута система: Windows, то платформа: ".Net" устанавливается вместе в пакете и не требует дополнительных действий.

На других операционных системах: Unix, MacOs, есть свои версии платформы: ".Net", которые позволяют запускать программный код, который написан на совместимых языках и на этих операционных системах.

## Особенности платформы .Net (dot Net)

* ".Net" (dot Net) - это свободная платформа (is a free) с открытым исходным кодом (open-source) для кросс-платформенной разработки (cross-platform development platform: Linux, MacOS, Windows);

**"is a free"** - означает, что платформа распространяется бесплатно. Продукты, которые созданы при помощи платформы, могут быть использованы по усмотрению разработчиков, в том числе и с целью коммерческой выгоды;
**"open-source"** - означает, что открытый исходный код доступен для анализа и предложения по улучшению исходного кода принимаются разработчиками платформы: ".Net" и рассматриваются и могут быть включены в последующие релизы платформы;
**"cross-platform"** - продукты создаваемые при помощи платформы: ".Net", можно запускать на операционных системах: Linux, MacOS, Windows.

* **Платформа** - это совокупность языков программирования, библиотек и сервисов;

Каждый отдельный компонент большой ценности не представляет. К примеру, язык программирования C# нельзя представить без основополагающих библиотек или библиотеки невозможно представить без соответствующих сервисов (управление памятью и сборка мусора). По отдельности это не работает, ценность заключается именно в совокупности.
Библиотеки с классами и функциями созданы с тем расчетом, что действия будут кем-то выполняться (сервисом).

* Область применения: предназначена для разработки разных типов приложений: Mobile, Desktop, Web, Cloud, Microservices.

Среда (платформа): ".Net" предназначена для масштабных приложений верхнего уровня (в основном) для различного рода автоматизации в разном форм-факторе.

Каждая технология - это набор библиотек. Для того, чтобы создать приложения с помощью WPF, нужно подключить библиотеку и использовать классы, которые там реализованы. Каждая технология подразумевает определенную идею, которой нужно пользоваться.
Создание приложение консольного совсем не так делается, нежели создание приложения оконного.
Создание приложения оконного на базе Windows Form совсем не так делается, как создание приложения на основе WPF, т.е. там именно идеология другая, а не в том, что классы разные.
К примеру, у оконных приложений логика работы основана на событиях (событийно управляемый код), а в консольных приложениях ничего такого нет.
По каждой технологии есть документация.

Если необходимо создать приложение на платформе: ".Net" и уже есть техническое задание и понимание, что нужно заказчику, то:
1. Нужно подобрать необходимые технологии;
2. Если технология незнакома, то почитать документацию и изучить ее;
3. Создать проект, подключить библиотеки и писать код с их использованием;

Общий список технологий платформы: ".Net" и их возможности необходимо знать.

* У многих технологий, есть общая основа - это компоненты, которые их  объединяют. К примеру, это доступ к данным. Данные приходится забирать из различных источников данных в приложениях разного форм фактора. 

Наверняка, web-приложение или оконное приложение необходимо подключить к источнику данных (реляционная БД, файл в формате: xml, css и др. с информацией и т.д.). Ранее, чтобы подключить приложение к источнику данных, необходимо было изучить всю спецификацию источника и взаимодействовать с источникам только по средствам тех или иных протоколов, которые он поддерживает. Сейчас, есть технология: LINQ и другие, которые помогают оперировать данными посредствам создания объекта и манипуляций им с помощью методов (получить, удалить, записать и т.д.), что является несомненным преимуществом, так как позволяет глубоко не вдаваться в подробности, что происходит на стороне источника.

* Base Class Library (BCL) - это библиотека классов, на самом деле - это не одна библиотека, а несколько, которые содержат описание типов (bool, string, int и др.), которые являются компонентами библиотеки.

На каком бы языке не пришлось писать: VB, C#, F#, объявляемые переменные будут привязаны к базовому классу, в котором описаны тип. Но наименования типов в языках программирования могут отличаться, но базовый класс, к которому они будут привязаны будет один и тот же (пример: в C# есть тип: float, а в VB он называется: single).
Все преобразования к универсальному типу происходят в процессе компиляции. К примеру в C# есть тип: "short", который называется в системном классе: "int16" (аналогичный пример с типом: "int" в C#,  в системном классе он называется: "int32")

Помимо Base Class Library (BCL), есть и другие классы, которые предназначены для конвертации переменных разных типов, для математических операций, доступа к файлам и т.д.

Независимо от того на каком языке пишется программа и какой тип проекта, классы используются одни и те же.

* Common language Runtime (CLR) - это вся механика, которая позволяет приложению быть откомпилированным и запуститься.

C 2014 года - ".Net Core" - платформа для разных ОС. Текущая версия - ".Net 8" (начиная с версии 4 без слова: "Core")

Вначале компания Microsoft придумала платформу: ".Net Framework", данная платформа была ориентирована только для разработку приложений для Windows.

Приложения, созданные на платформе: ".Net Framework" нельзя было откомпилировать, чтобы они запускались на других операционных системах, кроме Windows, а приложения, созданные на платформе: ".Net Core" можно было откомпилировать, чтобы они запускались на других операционных системах.
До 2019 года платформы ".Net Framework" и ".Net Core" существовали, как 2 разных продукта, с 2019 года компания Microsoft объявила, что переходит на универсальную open-source платформу: ".Net".

## CLR и JIT-компиляция


* **CLR (common language runtime)** - общеязыковая среда (виртуальная машина), которая отвечает за загрузку приложения, старт его и управление памятью.

Ранее, за управление ресурсами: память, драйверы, взаимодействие с устройствами и др. за это за все должен был отвечать программист, он выделял память, он обязан был следить за ее использованием и освобождением.
Типичные ошибки в приложениях на старых языках - это утечка памяти (memory leak). Особенно это чревато для общей работы приложения, когда это происходит в цикле или рекурсии. Другая опасность - это обращение не в нужную область памяти, а чужую - это чревато крахом и системными ошибками.

* Приложения с управляемым кодом (managed code).

Приложение старого формата, чаще всего были монолитными и ошибка в одной части программы, приводила к закрытию системы целиком. Сейчас архитектура современных информационных систем более гибкая и управляемая. Если вдруг, падает какой-то сервис, то это не приводит к краху всей системы, а просто не работает какая-то ее часть и у самой системы есть возможность определить ошибку и перезапустить сервис, т.е. обрабатывать ошибку управляемо.

Окончание: "ed" (manag**ed** code) - говорит о том, что это не наш код управляет чем-то, а нашим кодом управляют, т.е. наш код является пассивным элементом. Не мы управляем чем-то, а нами управляют. Данная ситуация была не очень понятна для "старожилов", так они сами привыкли выделять или освобождать память и т.д.
Если говорить о приложениях большого масштаба, то свобода выделения и освобождения памяти самостоятельно - это иллюзия, так как с данной свободой многократно возрастает и ответственность, ведь любая ошибка может привести к краху приложения или даже всей системы.  
Когда делегируем какой-либо структуре выделение и освобождение памяти, то это находится не под контролем, но об этом не нужно задумываться. В современном мире, большинство языков программирования работают именно по такой схеме:

* **GC – garbage collector (сборщик мусора)** - это автоматический менеджер памяти который периодически освобождает память, удаляя объекты, которые больше не используются программой. GC используется во многих современных языках программирования.


При запуске приложения происходит следующее:
1. Для приложения выделяются ресурсы, в первую очередь память;
2. В память выгружаются данные и сам программный код;
3. Управление передается программному коду;
4. Команды, которые записаны в программном коде - начинают выполняться, их выполняет специальный компонент, который называется **виртуальной машиной**;
Для приложения создается виртуальная машина и выполняет код. Выполняет - читает команду, интерпретирует ее и передает операционной системе на выполнение уже на языке самой системы. Таким образом, виртуальная машина является прослойкой между приложением и операционной системой.
Это дает кросс-платформенность, так как мы не обязаны в своем приложении разговаривать на языке операционной системы. Можем даже не знать, какая операционная система будет.
Задача виртуальной машины перевод кода в машинный код, для Windows своя виртуальная машина (среда выполнения) для MacOs - другая среда выполнения, но они все входят в ".Net", просто они разные для разных операционных систем.

Компания Microsoft принципиально изменила процесс перевода текста программы в машинный код.

В ".Net" используется средний подход между компиляцией и интерпретацией.
С одной стороны, действительно создается файл в формате: ".exe", который содержит команды не на родном языке. Все команды в нем уже переработаны компилятором в понятный и удобный для исполнения вид и там уже проанализированы все ошибки. 
С другой стороны, после компиляции исполняемый файл содержит не нативные команды операционной системы, а команды на специальном псевдоассемблере (макроассемблере), он в платформе: ".Net" называется: **IL (intermediate language)**.

**IL (intermediate language)** - это низкоуровневый язык, который с одной стороны, очень понятен и его не нужно переводить специально в машинный код, а с другой стороны, он независимый от платформы.

Задача виртуальной машины (среды выполнения), взять команду на языке **IL (intermediate language)** из исполняемого файла в формате: ".exe" и перевести данный код уже в машинную инструкцию и выполнить - этот процесс называется **JIT (just in time compilation)**.

## RID (runtime ID) идентификатор среды выполнения

У разработчиков на C# есть возможность переводить проект в переносимый формат или привязывать его к конкретной платформе (делать его зависимым от платформы).
Если привязывать проект к конкретной платформе, то его нельзя будет запустить на другой операционной системе, но приложение будет более быстрое и с большими возможностями для разработчика. В этом случае сама операционная система, а конкретно ее версия и разрядность записывается прямо в исполняемый файл и это называется **RID (runtime ID)**.

* **RID (runtime ID)** - идентификатор среды выполнения. Он есть только в случае, если проект непереносимый;
* RID содержит информацию о платформе ОС и архитектуре ЦП, на которых работает приложение ".Net"; 

*Пример*: 64-разрядная версия Windows, 64-разрядная версия Linux.

## Некоторые термины

**App model** - модель приложения (форм-фактор), это то что будет на выходе, как будет восприниматься пользователем, будет ли это вообще с пользователем взаимодействовать и каким образом все строится.
Примеры: ASP.Net, EF (entity framework), Windows Forms, WPF (windows foundation presentation)

**Assembly** – сборка, это часто исполняемый файл EXE или DLL файл, который может содержать одну или несколько наборов **API (Application Programming Interface)**, которые могут быть вызваны приложениями или другими сборками. Сборка находится в папке bin.

**API (Application Programming Interface)** - это программный интерфейс с помощью которого внешняя среда может взаимодействовать с сборкой. Это некий набор классов и в классах функций со своим названием и сигнатурой, который мы можем вызывать.

Для исполнительной среды, главное - это точка входа, т.е. это первая функция, которую мы можем вызывать и которая начинает работать и потом уже вызывает сама все остальное.
Для консольного приложения (среды выполнения) - это функция: "main". Для библиотек это весь набор классов и функций.

**Сборка (Assembly)** - это полноценный комплект файлов, необходимых для полноценной работы приложения или его использования.

**Ecosystem (экосистема)** - all of the runtime software, development tools, and community resources that are used to build and run applications for a given technology.
Под экосистемой понимается весь набор средств, который позволяет работать той или иной информационной системе. Это не только сам исполняемый файл (EXE), но и необходимые для его работы библиотеки, которые не только используются самой программой, но и те которые нужны для его запуска и исполнения. К примеру, классы для сборки мусора или сама виртуальная машина относятся к экосистеме. Иногда в экосистему могут входить средства, которые нужны в проекте, например, средства для обращения к базе данных или обращение к какому-то виду устройства.

В платформе: ".Net" компания Microsoft предлагает не какой-то один язык или библиотеки для него, а целую экосистему, которая работает бесшовно.

К примеру, есть проблема "стыковки", если приложение пишется на языке: Kotlin, PHP  и его нужно состыковать с базой данных Mongo DB или MS SQL Server. Из-за того, что продукты разрабатываются разными производителями, могут возникать проблемы в отсутствии схожих типов данных и т.д., которые не позволят быстро "состыковать" технологии.

Использование множества различных технологических решений от различных разработчиков - это альтернатива единой экосистеме разрабатываемой одной компанией или различными разработчиками под одним общим стандартом.

**.Net CLI (command-line interface)** - кроссплатформенное средство для разработки, компиляции, запуска и публикации ".Net"-приложений.

**Platform (платформа)**. Операционная система и аппаратная часть, на которой она работает. Например, Windows, macOS, Linux, iOS, and Android.

У слова: "платформа" несколько смыслов:
* Если говорить о платформе в контексте: ".Net", то это весь набор программных средств, разработанных компанией Microsoft для создания приложений;
* Если говорить о платформе в контексте компиляции приложения, то это связка операционной системы и аппаратной части, т.е. это какая разрядность операционной системы.

**Runtime** - исполняемая среда для управляемого приложения.

**Stack (стек)** - a set of programming technologies that are used together to build and run applications.
Стек - связка технических средств, которые используются для разработки приложения.

## Публикация приложения

Пользователю не нужны лишние манипуляции для запуска приложения, самый идеальный вариант - это ярлык приложения которое запускается при помощи ярлыка на рабочем столе.

Подробнее про публикацию приложения можно почитать на официальном сайте компании Microsoft или на каких-либо сторонних сайтах.

Для сборки проекта в Visual Studio, необходимо в верхнем меню нажать на подменю: "Сборка" и выбрать необходимую команду: "Собрать решение" или "Сборка %наименование приложения%".
При нажатии на команду: "Собрать решение" будут скомпилированы все отдельные проекты, которые входят в решение и которые могут быть написаны на различных языках программирования.

Компиляция происходит не быстро и это занимает значительную часть времени.

Можно запускать на компиляцию не все решение, а отдельные его проекты, если, к примеру, изменения были только в одном проекте, то можно скомпилировать только его, а для остальных будет использоваться старая версия. При этом решение придется пересобрать.

При сборе необходимо уточнять, что необходимо сделать с внешними библиотеками (не те которые написаны нами), всего 2 варианта:
* Обращаться к ним из специального репозитория. 
Библиотека лежит где-то в системной папке и достается кодом из нее. Это означает, что если наш исполняемый файл будет перенесен на компьютер пользователя, то мы ожидаем, что и у него на компьютере есть такая же библиотека, такой же версии. Это устанавливается из реестра библиотек, библиотека должна быть зарегистрирована в операционной системе по ее идентификатору (ID) она находится, но названия версий должны совпадать. 
При таком варианте, есть риски, что если библиотека не обнаружиться, то программа просто не запуститься.


* Альтернативный вариант, сформировать внешнюю папку, где разместить все необходимые библиотеки и тогда зависимость от наличия библиотек у пользователя пропадает.

Кроме всего, при сборке можно настраивать различные параметры для сборки.

В верхнем меню в подменю: "Сборка", есть команда: "Опубликовать выбранные элементы".
Публикация - это тоже самое, что и сборка, но плюс еще deployment решения на какую-то платформу, например, в облако или куда-то еще.
При использовании команды: "Опубликовать выбранные элементы", необходимо выбрать, куда именно нужно опубликовать приложение и задать необходимые при публикации параметры.

При публикации автономного приложения создается исполняемый файл, который включает среду выполнения и библиотеки .NET, а также приложение и его зависимости. Пользователи приложения могут запустить его на компьютере, на котором не установлена среда выполнения .NET. Автономные приложения зависят от платформы.

Автономная сборка хороша тем, что пользователю не нужно ни о чем задумываться. Передали пользователю папку с программой, он у себя ее запустил и все работает. Но размер программы очень большой, из-за копирования в папку с программой всех необходимых компонентов.

При публикации приложения в качестве зависимого от платформы создаются исполняемый файл и двоичные файлы (файлы DLL), включающие только само приложение и его зависимости. Пользователям приложения необходимо отдельно установить среду выполнения .NET. 

**В чем принципиальное отличие от запуска файла: ".exe" и ".dll"?!**

Есть 2 понятия файлов: ".dll" абсолютно разных и не нужно путать:
1. Есть понятие библиотеки в смысле шаблона проекта и при компиляции создается файл: ".dll" и там нет никакой функции: "main" и конечно же библиотеку не запустить через командную строку, библиотеку можно использовать другим программистам в другом проекте, подключаться к библиотеке и вызывать какие-либо функции;
2. Если проект создан на базе обычного консольного приложения, то при компиляции может создаться файл: ".dll" - это не будет библиотекой и данный файл можно запустить через командную строку, так как в нем будет функция: "main" (точка входа);


# Лекция Р3: Разработка проектов с использованием WinForms

## Объектно-ориентированный подход
Объектно-ориентированный подход - это была эволюция развития написания программы. Когда команд становится много и они сложны, то необходимо задуматься об организации программного кода, так как ориентироваться в коде без определенной структуры очень сложно (как убрать главы в книге, например).
Основным вопросом над которым задумались стал вопрос: А нельзя ли организовать код таким образом, как мы видим организацию объекта в окружающем мире. В окружающем мире мы сталкиваемся с осязаемыми сущностями - объектами (пульт, мышь), которые имеют определенные характеристики: длина, цвет, вес, количество кнопок, а также имеет определенные функции и события (например, индикатор топлива в автомобиле, он сообщает о недостатке топлива - инициатор события сам автомобиль). Так и появились понятия объекта и класса.
- Объект - это конкретная сущность с которой работаем в программе. 
- Класс - это тип объекта.
Объект наследует из класса, все, что там заявлено.
*Пример*: Можно объявить класс: "пульт" и сформулировать обязательные характеристики, которыми обладают все пульты, такие как: длина, ширина, цвет, вес. С другой стороны, пульт может иметь определенные специфические характеристики, которые может иметь только определенный пульт. Это очень похоже на статью в энциклопедии про тигра, статья будет классом, так как это некая абстракция, а реальный тигр, которого можно встретить в природе - это будет объектом класса, он будет иметь свой цвет, вес, возраст и т.д. - это объект созданный на базе класса.

Объект состоит из:
- Method - метод. Действие, применимое к объекту. Метод - это аналог функции, это действие, которое можно выполнить с объектом. Метод применяется к конкретному объекту. Пример: есть объект: "кнопка" и необходимо ее передвинуть, метод будет: "move", а записать можно: Button.Move.
- Event - событие. Информирование владельца объекта. Способ уведомления владельца объекта о том, что с ним произошло. 
- Property - свойство. Характеристика объекта. Также как и переменная имеет свое название и тип. В отличие от простых переменных, свойство не может быть само по себе (нельзя написать x = 100), нужно написать название объекта и имя свойство через точку (пример: Object.X).

## Понятие массива

**Массив** - это именованная область памяти, состоящая из однотипных, проиндексированных элементов.

*Пример одномерного массива*:
![[Pasted image 20260220000404.png]]

*Пример синтаксиса объявления одномерного массива*:
```C#
[public | internal | private] type[] arrayname;
 
[public | internal | private] type[] arrayname = new type[size];

// size – это количество элементов в массиве
```

***Важные моменты***:
1. Все ячейки находятся в памяти в одном блоке, ячейки соседние и не могут находится в разных местах в памяти;
Эта особенность позволяет обратиться к ячейке зная ее индекс. К примеру, если необходима ячейка под индексом: "7", то "int" (4 байта) умножаем на 7 получим 28 и смещаемся от начала на 28 байт и получим значения в ячейке с индексом: "7".
Таким образом обращение к ячейке массива происходит за единицу времени, но это не всегда так, к примеру, в списках так не получится сделать, там недостаточно умножить индекс на размер, так как там ячейки могут находится где угодно и не обязательно в соседних ячейках.
Минус массива: если необходимо вставить ячейку между двумя соседними ячейками, то так просто не получится сделать, придется выполнять операции копирования массива в соседнюю ячейку памяти.
2. Индексы. На сомом деле, никаких индексов в памяти нет, есть просто память выделенная для ячеек и там хранятся значения (массив числовой - хранит числа). Индексы - это некоторая условность.
Индексы есть только в понимании программиста (понимании алгоритма) и когда мы обращаемся к ячейке массива a\[7], то это служит информацией для компилятора, что нужно сделать (нужно взять размер ячейки и умножить на 7, таким образом посчитать на сколько байт сдвинуться и обратиться в нужное место). Обычно, современные компиляторы все это делают заранее.

Формула: offset = index * size
Где: 
- offset – смещение в байтах
- index – индекс элемента
- size – размер одного элемента в байтах.

3. Ячейки однотипные. Требование однотипности, гарантирует, что размер всех ячеек одинаковый и мы можем посчитать смещение. В этом есть сила массива.

Вопрос: Если ячейки типа: "string" (которые имеют различную длину), то как компилятор может корректно посчитать куда сместиться?
Массив с типом: "string" содержит ссылку на область памяти, в которой уже хранятся коды символов и столько символов, сколько нужно. Это не противоречит требованию однотипности ячеек, а ссылка на тип: "string" одинакового размера.

*Примечание*: Ссылки на объекты тоже могут иметь разный размер

*Преимущество использование массивов*:
- Структура представления данных;
- Экономия сил программиста;
- Эффективный обход и обработка элементов;
- Динамические массивы;

## Массив - переменная

**Массив - это переменная**, у нее есть имя и все характеристики, как у переменной, переменная не обычного типа, а объектного.

*Характеристики переменной*:
- Имя;
- Область видимости;
- Время жизни;

Массив можно объявлять, как и простые переменные, внутри функции, тогда они будут локальные или внутри цикла/блока, можно объявить за пределами функции (private/public), тогда массив (переменная) будет либо закрытая, либо публичная (видна везде).

**Отличия массива от переменной**:
1. Размер. 
Когда мы говорим: "Размер переменной", то имеем в виду, сколько памяти для переменной выделено (пример: размер переменной с типом: "int" = 4 байта).
Размер массива, не совсем корректно считать, как размер переменной. Массив - это переменная ссылочного типа, она хранит ссылку. Ссылка занимает тоже какую-то память, но имеется в виду, область памяти, которую занимают ячейки массива. Чаще всего, когда говорят: "размер массива", то подразумевают размер не в байтах, а количество ячеек массива.
2. Размерность.
Данная характеристика, характерна только для массива и она определяет, как ячейки организованы и как к ним осуществляется доступ.

*Пример*: Если взять 100 кубиков (100 ячеек), то как их можно выложить на столе?
Самый простой вариант: выложить в линию и это будет соответствовать одномерному массиву.
Можно выложить кубики в виде прямоугольника, размером 5х20 или 4х25 или 2х50 - это примеры двухмерных массивов, у которых размер одинаковый, но конфигурация разная.
Можно привести пример трехмерного массива (слои), выстраивается в объемную фигуру.

*Примечание*: В памяти компьютера никаких кубиков, которые так выстраиваются - нет, в памяти компьютера, массивы могут быть 4-х мерные, 5-ти мерные, 6-ти мерные и т.д. Привязываться к геометрии не стоит, эта аналогия только для первичного понимания особенностей организации данных.

Размерность определяет способ доступа к ячейке, какова размерность, столько индексов и необходимо указать для доступа к ячейке. Если массив 4-х мерный, то необходимо указать 4 индекса (4 координаты), чтобы в 4-х мерном пространстве обратиться к конкретной точке, в 3-х мерном - 3 координаты (x,y,z), в 2-ух мерном 2 координаты (x,y).

Число ячеек зависит от того, сколько данных необходимо хранить и если данных 100, то нельзя объявить размер другого массива. Теоретически, все массивы могут быть одномерные, вопрос только в том, как удобнее.

Индексация ячеек начинается от: "0" (нуля), а при объявлении массива, указывается количество ячеек и поэтому, максимальный индекс отличается от количества на "1" (единицу).

3. Ссылочный тип. 
Массив - это переменная ссылочного типа, она хранит ссылку на область памяти, где находится значение.

## Особенности массивов в C\#

В классических языках (C или Pascal) нет классов, есть понятие массивов и это не связывается с термином класса.

**Важно**: Отличие объектов, созданных на базе класса, от простых переменных, созданных на базе обычных типов в том, что объект нужно не просто объявить, а еще и создать при помощи слова: "new" (это один из вариантов, он базовый).

Массив можно пересоздавать (создать -> уничтожить -> создать снова), с простыми переменными так не получится.

- В качестве индекса можно указывать переменную;

Когда необходимо написать функцию, которой необходимо передать массив, то указывается аргумент необходимым образом, то подразумевается, что он существует и уже заполнен и работа идет с данным массивом. 
При этом данные о массиве, получать необходимо через его свойства.

```C#
public static void SumArray(int [] arr)
{
	int sum = 0;
	foreach(int n in arr) sum += n;
	return sum;
}
```

**Важно**: 
- Так как массив ссылочный тип данных, то при изменении значения ячейки в функции, в массиве значение поменяется.
- "ref"  в аргументе в функции нужно писать тогда, когда в ней меняется сам массив, к примеру, в функции пересоздается массив, вызов конструктора или изменении размера (resize).


## Динамический массив

Термин динамический массив существует с давних времен. В старых языках программирования можно было объявлять динамические и статические массивы;

**Статический массив** - это массив, размер которого в течение работы программы не меняется.
**Динамический массив** - это массив, размер которого в течение работы программы можно менять.

В старых языках программирования, динамические массивы были сопряжены с некоторыми сложностями на уровне компилятора с ними сложнее строилась работа.
В современных языках программирования, разделение на динамические и статические массивы ушло.

В C#, массив - это объект класса Array, а у класса Array размер (число ячеек) можно менять. Нельзя объявить так массив, чтобы его размер невозможно было изменить, если только не запрограммировать свой класс или использовать другой класс.

Для изменения размера массива в C# используется метод: Array.Resize()

У метода: Array.Resize() есть ограничение, он позволяет изменять размер только для одномерного массива. 
Если необходимо изменить размер у двухмерного массива, то необходимо самим это запрограммировать, для этого нужно:
- объявить новый массив другого размера;
- скопировать данные из старого массива в новый;

*Пример использования метода Array.Resize*:
```C#
//Абсолютный вариант (если массив больше, то данные потеряются)
Array.Resize(ref arrNums, 4);

//Относительный вариант (отталкиваемся от текущего размера массива)
Array.Resize(ref arrNums, arrNums.Length + 5);

```

Так как Array.Resize операция очень трудоемкая (выделение новой области в памяти и копирование) и каждый раз вызывать метод: Array.Resize и добавлять увеличивать размер массива на 1 ячейку - это не оптимальный вариант.

*Рекомендация*: Если читаются данные из файла и неизвестно, какого размера необходимо создать массив, то можно использовать следующие варианты:
- Если есть возможность сделать предварительную оценку "сверху", то необходимо изменить размерность массива, исходя из оценки, а не заполненные ячейки потом убрать;
- Если возможности сделать оценку "сверху" нет, то необходимо увеличивать размер массива с определенным шагом, если останутся незаполненные ячейки, то потом их убрать;


## Некоторые операции с классом Array

```C#
//Копирование массива
array1 = array2;

//Копирование значений массива (массивы должны иметь одинаковую размерность, array2 должен иметь достаточное количество ячеек)
Array.Copy(array1, array2, 4);

//Клонирование массива
array2 = (int[,])array1.Clone();

//Изменение размера массива (только для одномерных массивов)
Array.Resize(ref array1, 10);
```

Статические члены класса - это такие члены класса (либо метод, либо свойство), которые вызываются не применительно к объекту, а применительно к классу (**Array**.Resize).

При использовании оператора присвоения "=" применительно к переменной, то значения одной переменной, копируется в другую переменную (x = y).

При использовании оператора присвоения "=" применительно к массиву, происходит копирование ссылки, а не копирование ячеек.

Прямого метода, который бы гарантировал клонирование массивов нет, потому, что глубина вложенности ссылок может быть большая. В нашем массиве, ячейка может содержать ссылку на другой массив, а тот массив может содержать в ячейках ссылки на какие-то другие массивы и т.д.

Поскольку массив ссылочного типа, то операция:
```C#
//Копирование массива
array1 = array2;
```
Дает не то, что кажется на первый взгляд. В переменных обычного типа, при операции присвоения происходит копирование данных из одной переменной в другую.
Если мы имеем дело с массивами, то в переменной имеется ссылка на какие-то ячейки в которых находится информация. Если мы пишем array1 = array2, то скопируется ссылка на ячейки и в переменной array1 будет записана ссылка, которая ссылается на ячейки памяти, на которые также ссылается переменная array2.
Если поменять ячейки через array1, то эти же значения будут доступны и в array2, так как обе переменные ссылаются на одну область памяти.
Если необходимо создать копию массива, то необходимо использовать следующие методы: Array.Copy() или Array.Clone().
Отличие метода: Array.Copy() от Array.Clone() в том, что метод: Array.Copy() - статический, а метод: Array.Clone() не статический. Метод: Array.Copy() вызывается чрез название класса, а Array.Clone() через имя объекта: array1.Clone(). Еще есть разница в работе данных методов и она заключается в том, когда ячейки массива являются ячейками ссылочного типа, в этом случаи методы ведут себя по разному. 
Для массивов обычных (скалярных) типов (например, int), разницы в том, какой метод использовать: Array.Copy() или Array.Clone() - нет.

## Ступенчатый массив (jagged array)

**Ступенчатый массив** - это массив каждая ячейка которого содержит ссылку на другой массив и не обязательно, что размерность вложенного массива будет совпадать с "основным".

Создать за 1 раз ступенчатый массив не получится, вначале создается базовый массив, а далее для каждой ячейки создается свой массив.

*Синтаксис ступенчатого массива*:
```C#
array[][] = arrJag;
arrJag = new int[3][]; // базовый массив
arrJag[0] = new int[10];
arrJag[1] = new int[20];
arrJag[2] = new int[30];
```

Когда ступенчатый массив не нужен, это когда есть массив с информацией об оценках студентов, то это типичный двухмерный массив, по строкам фамилии студентов, по столбцам названия предметов, на пересечении оценка конкретного студента за конкретный предмет.

Ступенчатый массив может подойти, если мы хотим хранить информацию о температуре, замеряемой каждый день 1 раз в разрезе месяцев. В таком случае, в каждом месяце различное количество дней и обращаться к данным будет удобно.

В силу некоторых сложностей, когда можно отказаться от ступенчатого массива, лучше от него отказаться. Альтернатива, к предыдущему примеру, все же сделать массив двухмерный 12х31

## Алгоритмы поиска

***Основной смысл***: Задача поиска заключается в нахождении элемента в некой группе элементов (массиве, списке и т.д.) с заданным значением, это называется значением ключа. Ключ не всегда равен значению элемента.

***Возможные варианты алгоритма***:
1. Нашли элемент (возможные варианты):
	- Ищем первый подходящий элемент = возвращаем его индекс
	- Ищем все подходящие элементы...
2. Не нашли элемент

Функция получает на вход: массив и значение ключа
Функция должна вернуть на выходе индекс или индексы найденных элементов.
Если элемент не найден, то существует 2 варианта, что можно вернуть функции:
1. Особое значение, которое заведомо не может быть ответом. Например, значение: "-1" при поиске значений типа: "int", так как ячейки с индексом: "-1" не бывает;
2. Функция возвращает значение bool (true/false), а индекс возвращается через выходной аргумент (out);

Существуют несколько вариантов (идей) решения задачи поиска:
- Линейный поиск - 

### Линейный поиск
Самый банальный, базовый вариант. Перебор ячеек от первой и до какой:
- Если в условии указано, что необходимо найти первое встреченное значение, совпадающие с ключом, то цикл должен быть до момента нахождения значения;
- Если в условии указано, что необходимо найти все встреченные значения, совпадающие с ключом, то цикл должен отработать до самого конца;

Дано: одномерный массив int[] фиксированного размера, значение ключа 
num
Выход: признак успешности, индекс

```C#
// Пример функции Линейного поиска элемента 
static bool LinearSearch(int[] arrItems, int num, out int index)
{
	for (index = 0; index < arrItems.GetLength(0); index++)
		if (arrItems[index] == num) return true;

	index = -1;
	return false;
}
```


#### Линейный поиск, несколько вхождений

Если необходимо найти не только первое вхождение (или все вхождения) элемента в массив, то существуют следующие варианты организации функции:
1. Создать массив, в который записывать все индексы найденных ячеек и этот массив вернуть. 
Есть проблема, которая заключается в том, что заранее невозможно узнать, сколько элементов будет найдено, чтобы ее решить 
- Нужно задавать массив с запасом, т.е. равный по размеру входного массива, а потом перед выходом из функции уменьшим массив методом: "Array.Resize(ref array, newSize)" до нужного размера;
**ЛИБО**
- Изначально массив пустой, как нашли элемент, увеличиваем массив на 1 и записываем индекс найденной ячейки, потом пробегаемся по циклу дальше и если что-то еще найдено, то опять увеличиваем массив на 1 и записываем индекс;
Какой из способов предпочтительнее, на этот вопрос нет однозначного ответа. Если оценивать по времени, то первый способ лучше, один раз создали массив и записываем в него данные, а изменить размер нужно будет только 1 раз в самом конце. Это однозначно будет быстрее.
Но метод: "Array.Resize(ref array, newSize)" (изменение размера массива) - это ресурсоемкая операция, так как когда компьютер обрабатывает метод, то ему нужно выделить память, скопировать все ячейки из одной области памяти в другую и освободить память которая больше не будет использоваться.
Получатся, если массив огромный, состоит из 1 млн. ячеек, то нам нужно объявить массив на 1 млн. ячеек и там, к примеру, есть 1 вхождение, то мы запишем 1 значение в массив из 1 млн. ячеек и в конце будем использовать метод: "Array.Resize(ref array, newSize)" на массиве из 1 млн. ячеек, тогда вопрос будет зависеть уже от памяти.

```C#
// Пример сигнатуры функции Линейного поиска элемента с несколькими вхождениями
static int[] LinearSearchMult(int[] arrItems, int num)
{
 // Тело функции
}
```

2.  Регулирование числа вхождений с помощью дополнительного входного аргумента - номер стартового элемента для поиска

Для реализации такой идеи, нужно добавить дополнительный входной аргумент: "start".  По умолчанию в самом начале работы, он будет равным значению: "0". Он обозначает с какой позиции необходимо начинать поиск.

Функция ищет нужное значение, находит, останавливается на месте, через выходной аргумент: "index" возвращает номер значения вызывающей стороне. #### Линейный поиск, с барьерным элементом

Операцию линейного поиска элемента можно улучшить (модернизировать), если отказаться от одной операции в цикле при проверке.

```C#
// Пример функции Линейного поиска элемента 
static bool LinearSearch(int[] arrItems, int num, out int index)
{
	for (index = 0; index < arrItems.GetLength(0); index++)
		if (arrItems[index] == num) return true;

	index = -1;
	return false;
}
```

В цикле: "for" 4 операции:
1.  Сравнение элемента ячейки с искомым ключом:
```C#
if (arrItems[index] == num) return true;
```
От данной операции отказаться нельзя, так как основное условие поиска

2. Сравнение элемента счетчика с границей массива:
```C#
index < arrItems.GetLength(0);
```
Данная операция предостерегает от выхода за границы массива, от нее можно попробовать отказаться, путем добавления барьерного элемента в массив.

3. Увеличение счетчика:
```C#
index++
```
От данной операции отказаться нельзя, так как она гарантирует проход по всем элементам в массиве.

4. Операция возврата к началу цикла: for (невидимая операция, которая зашита в машинном коде)
```C#
for (index = 0; index < arrItems.GetLength(0); index++)
		if (arrItems[index] == num) return true;
```
От данной операции отказаться нельзя, так как она позволяет вернуться в начало цикла с новыми значениями.

Для того, чтобы отказаться от операции: "Сравнение элемента счетчика с границей массива", необходимо в конец массива добавить элемент с искомым значением.

Основная идея состоит в том, что есть одномерный массив, в котором мы ищем ячейку с необходимым значением. Мы добавляем в данный массив одну ячейку с необходимым значением, это означает, что мы в любом случае найдем искомое число в границах массива, либо мы найдем настоящее число, которое есть в массиве, либо найдем добавленный нами элемент. Поэтому, когда сработает условие выхода, нам необходимо выйти из цикла, а не из функции и проверить индекс ячейки, если это индекс ячейки самый последний, то это означает, что ничего не найдено и нужно вернуть "false", а если индекс не самый последний, то необходимо вернуть: "true".

```C#
// Пример функции Линейного поиска с барьерным элементом 
static bool LinearSearch(int[] arrItems, int num, out int index)
{
	// Получаем размер массива
	int len = arrItems.Length - 1;
	int lastItem = arrItems[len];
	arrItems[len] = num;
	
	while (arrItems[index] != num) index++;
	
	if (index != len)
		return true;
	else if (lastItem == num)
		return true;
	else
	{
		arrItems[len] = lastItem;
		index = -1;
		return false;
	}
}
```

Даже линейный поиск с барьерным элементом не самый оптимальный вариант для больших массивов, он больше подходит для маленьких массивов.
Если массив произвольный, то ничего другого придумать нельзя, а вот если предварительно с массивом поработать, то можно добиться оптимальной работы.

#### Оценка сложности линейного поиска

- Худший случай O(N). Где N размер массива
- Лучший случай o(1)
- Средний случай:
	- n - размер входных данных;
	- m - число групп;
	- Pi -вероятность того, что входные данные принадлежат группе с номером i;
	- ti - время, необходимое алгоритму для обработки  данных с номером i;

### Двоичный поиск
Распространенный подход в алгоритмизации, когда необходимо сделать некоторые действия, данные нужно обработать перед выполнением действия, а потом заниматься основной целью алгоритма.

Перед поиском необходимого элемента, данные в массиве можно отсортировать.
В упорядоченном массиве при поиске значения не нужно делать последовательный перебор элементов.

Для поиска элемента в отсортированном массиве, можно воспользоваться бинарным поиском - это когда мы отбрасываем часть элементов, которые точно больше или меньше искомого значения. Сравниваем постоянно с средним значением в массиве.

**Основная идея состоит в том**, что сначала массив нужно отсортировать (возможно уже подается отсортированный массив на вход), затем нужно вытащить средний элемент, посмотреть на него и отбросить часть элементов, в которых точно не может быть искомого значения и делать так до тех пор, пока не найдем искомый элемент.

```C#
// Пример функции Двочного Бинарного поиска
static bool BinarySearch(int[] arrItems, int num, out int index)
{
	// left - левая граница из просматриваемой части
	// right - правая граница из просматриваемой части
	// mid - номер среднего элемента
	int left, right, mid;
	left = 0;
	// Устанавливаем правую границу поиска
	right = arrItems.GetUpperBound(0);

	// Условие left <= right работает до того момента, пока стрелочки не поменяются местами 
	while (left <= right)
	{
		mid = (left + right) / 2;
		if (arrItems[mid] == num)
		{
			index = mid;
			return true;
		}
		else if (arrItems[mid] > num)
			right = mid - 1;
		else
			left = mid + 1;
	}
	index = -1;
	return false;
}
```

*Примечание*: 
В учебниках по алгоритмизации можно увидеть, когда одно условие проверки в цикле: while, а не два, как показано в примере. Условие: "arrItems\[mid] == num" (проверки на равенство) можно убрать, при таком случае, стрелочки left и right будут смещаться, пока не остановятся на одном элементе, после выходим из цикла и проверяем элемент, равен ли он num или не равен и возвращаем: True или False в зависимости от полученного результата.
Если так реализовать, то с одной стороны в цикле на 1 проверку будет меньше (if останется один), с другой стороны количество итераций в цикле будет одно и тоже, независимо от расположения данных, количество итераций в цикле будет одинаковое. В приведенном примере - это не так.

#### Оценка сложности  двоичного поиска

- Количество бинарных разбиений O(log2 N). Количество итераций равно количество делений множества элементов на равные доли, сколько можно разделить на 2 равные половинки. К примеру, если элементов 16, то понадобиться всего 4 итерации;
- Бинарный поиск:
	- + (плюс) Ассимптотика лучше;
	- - (минус) Нужно сортировать
- Линейный поиск
	- - (минус) Ассимптотика хуже;
	- + (плюс) Не нужно сортировать;

- Размер массива. Нужно ли в будущем изменять данные. Сколько раз надо выполнять поиск.

#### Графики сравнения Линейного и Двоичного (бинарного) поиска

При использовании линейного или двоичного (бинарного) поиска на небольших массивах разница не так заметна , то, к примеру, на массиве из 32 значений при использовании линейного поиска в худшем случае  будет 32 итерации, а при использовании двоичного (бинарного) поиска в худшем случае будет 5 итераций.

### Какой алгоритм поиска: Линейный или Двоичный (бинарный) лучше использовать на практике

Указанный выше график не означает однозначного превосходства использования алгоритма бинарного поиска на практике.
Для двоичного (бинарного) поиска работа возможна только с отсортированным массивом, а для линейного с любым.
Если предположить, что алгоритмам даны одинаковые условия и на вход подается массив, который еще нужно отсортировать для двоичного (бинарного) поиска, то ко времени его работы, необходимо прибавить время требуемое на сортировку массива, в это суммарно может быть больше, чем требуется времени для линейного поиска.
С другой стороны, если есть массив, который уже отсортирован и требуется выполнить по нему несколько операций поиска, тогда двоичный (бинарный) поиск выглядит предпочтительнее.
Если предположить, что ячейки в массиве вводятся, то их сразу можно вставлять в необходимом порядке и никаких действий при сортировке не потребуется, если сразу формировать упорядоченный массив.
С третьей стороны, нужно смотреть не только на число итераций, но и сколько секунд (микросекунд) нам понадобиться для поиска. Сколько занимает времени один проход по циклу, и на маленьких массивах, может оказаться, что линейный поиск будет работать быстрее, даже с учетом того, что итераций он делает больше. 
Но в любом случае наступит ситуация, при определенном размере массива, при которой двоичный (бинарный) поиск будет предпочтительнее, поэтому, нужно дать оценку длине массива и понять от какой длины массива бинарный поиск будет предпочтительнее.


### Интерполяционный поиск

**Интерполяционный поиск** - это усовершенствованный алгоритм поиска в отсортированном массиве, который, в отличие от двоичного (бинарного) поиска, оценивает положение искомого элемента, основываясь на его значении, а не только делит область пополам.

Есть разновидность двоичного (бинарного) поиска - это интерполяционный поиск. Он такой же как двоичный (бинарный) поиск, но в нем ячейка, которую мы вытаскиваем берется не из середины, а используется более хитрая формула, которая учитывает, какие значения в первой ячейке, какие во второй и что именно нужно найти. К примеру, если первая ячейка содержит: "0", а последняя: "101", а ищем значение: "99", то среднее значение (mid), нужно взять ближе к правому концу.

***Общий смысл***: чем больше отбросим ячеек, тем меньше останется, чем меньше останется, тем быстрее найдем.

### Выборка

**Алгоритм выборки** – это поиск элемента из множества, обладающего некоторым специальным свойством (свойствами), при этом его значение (ключ) не известно.

К алгоритму выборки относится: поиск минимального элемента, поиск максимального элемента, поиск K-ого по величине и т.п.

К примеру, бывают задачи, когда нужно найти не максимальный элемент, а второй за максимальным (второй по величине).

```C#
// Пример поиска максимального элемнта в массиве
static int GetMax(int[] arrItems)
{
	int max = arrItems[0];
	int index = 0;
	
	for (int j = 1; j < arrItems.Length; j++)
		if (arrItems[j] > max)
		{
			max = arrItems[j];
			index = j;
		}
	return index;
}
```

*Вопрос*: Какой имеет смысл передавать массив, отсортированный или не отсортированный?
Когда массив отсортированный - этот алгоритм не имеет смысла, потому что в отсортированном массиве, максимальный элемент - это последний элемент или первый, если он отсортирован по убыванию.

Обычно, алгоритм выборки предназначается для случайно сформированных массивов.

***Основная идея***: На вход функции подается случайно сформированный массив. Объявляются 2 дополнительные переменные, одна хранит текущее максимальное значение (max), другая - индекс текущего максимального значения (index). Самый грамотный вариант инициализации переменных, в качестве первых значений взять стартовую ячейку. А цикл по проходу массива, уже начать со второго элемента в массиве. В цикле сравниваем текущую ячейку с максимальной и если необходимо, то переопределяем переменные max и index. Иных вариантов, кроме того, как пройти по всем элементам массива не существует.

**Временная сложность алгоритма**: O(N)

## Алгоритмы сортировки

Сортировка (или упорядочивание) - это перестановка элементов внутри некоего множества таким образом, чтобы каждой пары элементов выполнялось правило их отношения.
Соответственно, это правило должно быть задано. Если для элементов определенного типа данное правило задано быть не может, то и сортировка тоже невозможна.

Правила отношения - это правила сравнения, т.е. это ответ на вопрос, какой элемент больше. Для пары элементов, правила отношения, сообщают, какой элемент больше или меньше или они равны. 

Для некоторых типов данных, правило отношения по умолчанию не задано. К примеру, массив типа структур или объектов - это не определено.
Если написать функцию, которая сравнивает 2 типа таких переменных и возвращает значение, то и структуры тоже могут быть отсортированы, но если такой функции не существует, тогда сортировка невозможна.

Алгоритмов сортировки очень много. Есть условное деление алгоритмов на:
- Простейшие - элементарные алгоритмы, где временная сложность: O(n^2)
- Более быстрые алгоритмы, где временная сложность: O(n * logN)

В интернете есть сайты, где собраны алгоритмы сортировки, также на некоторых представлена анимация процесса сортировки. Алгоритмов сортировки больше 10.

В алгоритмах сортировки нет ответа на вопрос, какой алгоритм лучше использовать. Есть статьи в интернете, где проводятся исследования по данному вопросу. 
По итогам проведенного исследования алгоритмов сортировки, применяемых на массивах одинакового размера, но разными ячейками были получены следующие результаты:
- Если данные в массиве заполнены случайно, то выигрышными оказываются алгоритмы одни; 
- Если данные заполняются не случайным образом, то могут оказаться выигрышными другие алгоритмы;
К примеру, если подать массив на вход алгоритму уже отсортированный или почти отсортированный, то некоторые алгоритмы не меняют скорость своей работы, т.е. время их работы не зависит от того, отсортирован массив или нет, зависит только от числа ячеек. Если массив отсортирован, то число итераций будет такое же, хотя если посмотреть на данные и понять, что они уже отсортированы, то алгоритм сразу бы остановился.

Как не запутаться в алгоритмах сортировки, которых очень много и они имеют различные названия в разных источниках. Есть основные алгоритмы сортировки с оригинальной концепцией, а есть их разновидности и если считать разновидности алгоритмов сортировки, то их будет несколько десятков.
Пример: Есть алгоритм линейного поиска, а есть алгоритм линейного поиска с барьерным элементом и рассматривать его как отдельный алгоритм, вопрос философский. С одной стороны, это тот же самый линейный поиск, с такой же асимптотикой O(N), но с другой стороны в алгоритме линейного поиска с барьерным элементом применены существенные изменения, изменения есть в массиве и в самом алгоритме.

Сказать о том, какой алгоритм сортировки самый лучший, крайне затруднительно, потому что у алгоритмы показывают разные результаты в зависимости от ситуации.
Оценка алгоритмов не всегда делается в секундах, иногда принимают во внимание другие факторы, например, простота и прозрачность алгоритма. Реальную разницу в работе программы, где сортируются массивы из 100 ячеек нельзя почувствовать, что будет быстрый алгоритм, что медленный, но медленный алгоритм будет проще в плане написания отладки и поддержки кода.

### Сортировка методом выбора

**Основная идея алгоритма**: массив условно делится на две части: отсортированную и пока не отсортированную. В самом начале, считается, что отсортированной части нет, весь массив целиком он не отсортирован. На графическом рисунке (находится ниже), отсортированная часть выделена желтым цветом.
*Примечание*: многие алгоритмы действуют по такому же принципу, когда массив условно делиться на 2 части.

В алгоритме выполняются следующие операции:
1. Заводим некий счетчик (индекс) и устанавливаем его равным первому элементу в неотсортированной части;
2. Начинаем цикл, в котором выполняем поиск минимального элемента в неотсортированной части. Чтобы найти минимальный элемент, необходимо пройтись в цикле по всем элементам;
3. Найденный минимальный элемент меняется местами с текущим элементом;
4. После обмена, неотсортированная часть массива уменьшается на один элемент. Если количество оставшихся элементов больше одного, то происходит переход на шаг 1. В противном случае выполняется выход из алгоритма.

В данном алгоритме сортировки используются 2 цикла:
- Первый цикл. Основной, он запускает весь алгоритм с 1 по 4 шаги несколько раз;
- Второй цикл. Внутри основного. Он делает проход для поиска минимума;

***Схема работы алгоритма***:
1. Основной цикл:
- Подготовка к обходу;
	2. Второй цикл: 
	- Обход элементов для поиска минимума;
- Обмен местами минимума с текущим элементом;
- Уменьшение неотсортированной части;

Все простые алгоритмы сортировки содержат 2 цикла, один внутри другого.

В основном (внешнем цикле) количество итераций будет: N-1.
Как это определить. За одну итерацию цикла, минимальный элемент ставиться на свое место. Первая итерация, поставили первый элемент, вторая итерация второй, третья - третий и т.д. Для массива из 10 ячеек, необходимо 9 итераций, 10 итерация не нужна, так как там остается один элемент и он по определению будет на своем месте.

Во втором цикле (внутреннем цикле), количество итераций будет разное, поскольку, это проход по ячейкам неотсортированной части, но походу сортировки он уменьшается. К примеру, вначале будет 10 итераций, потом 9 итераций, потом 8, 7, 6, 5, 4 и т.д.

Для расчета общего числа итераций, необходимо число итераций в основном цикле умножить на число итераций во втором цикле.

Общее количество итераций = кол-во итераций внешнем цикле \* кол-во итераций во внутреннем цикле.

**Временная сложность этого алгоритма: O(N^2) - верхняя оценка.** 

У всех простых алгоритмов сортировки, аналогичная верхняя оценка  временной сложности: O(N^2).
На практике конечно оценка временной сложности будет меньше, так как это оценка исходит из того, что внешний цикл - это N-итераций и внутренний цикл тоже N-итераций, но во внутреннем цикле точно будет меньше итераций.

Если необходимо точно посчитать оценку сложности конкретного алгоритма на конкретном массиве и сравнить работу данного алгоритма с другим, то в литературе принято считать количество сравнений элементов. К примеру, когда ищем минимум, то прибавляем счетчик сравнений.

В цикле ищем минимальный элемент (красная стрелочка). Запоминаем его индекс.
Меняем местами числа текущего элемента (синяя стрелочка) и минимального элемента, который был найден (красная стрелочка).

Почему сортировка называется методом выбора? Потому что мы из всех былых ячеек выбираем минимум, вытаскиваем и ставим его на свое место. От английского: Selection sort.  

### Сортировка методом прямой вставки

**Основная идея алгоритма**: массив условно делится на две части: отсортированную и пока не отсортированную. Но в данном алгоритме, в самом начале, считается, что в начале работы алгоритма отсортированная часть состоит из одного элемента.
Далее берем из не отсортированной части первый элемент и пытаемся его вставить в нужное место в отсортированной части.
Данная идея очень похожа, как человек раскладывает карты, которые есть у него на руке.

**Отличие алгоритма: "Сортировка методом прямой вставки" от "Сортировка методом выбора"**:
В алгоритме сортировки методом выбора, мы в не отсортированной части брали минимальный элемент, а не первый попавшийся, его еще нужно было найти. 
А в алгоритме сортировки методом прямой вставки, мы из не отсортированной части берем первую ячейку, но в какое место ее необходимо вставлять - это вопрос, для этого организуется внутренний цикл, который смотрит, в какое место необходимо вставить первую ячейку.

***Схема работы алгоритма***:
1. Основной цикл:
- Берем необходимую ячейку;
	2. Второй цикл: 
	- Обходим отсортированную часть (снизу-вверх) и смотрим, куда вставить ячейку;
	- Сдвигаем проверяемые значения на 1 вниз, если проверяемое значение меньше сравниваемого; 
- Вставляем ячейку в необходимый диапазон;
- Уменьшение неотсортированной части;

*Графический пример сортировки методом прямой вставки*:
![[Pasted image 20260410005516.png]]

Считаем, что массив уже частично отсортирован. По отсортированной области (выделена желтым) идем снизу -> вверх и смотрим, куда вставить первую ячейку из не отсортированной области.
Удобно, когда делайте цикл, организовывать сдвиг ячеек. Если проверяемые ячейки в отсортированной области больше текущего (проверяемого) значения, то сдвигать такие ячейки на 1 индекс вниз.

***У алгоритма сортировка методом прямой вставки, есть 2 особых случая***:
1. Если мы взяли число из не отсортированной области и дошли с ним до самого верха.
В данном случае, число нужно поставить на самое первое место в массиве. Эту ситуацию нужно предусмотреть, так как можно выйти за границы массива при работе алгоритма.
2. Взятое число из не отсортированной области никуда двигать не нужно;

В работе алгоритма сортировка методом прямой вставки, возможны 3 варианта:
1. Взятое число из не отсортированной области, необходимо вставить куда-то в середину отсортированной области;
2. Взятое число из не отсортированной области, необходимо вставить на самый верх;
3. Взятое число из не отсортированной области, никуда двигать не нужно;

Поскольку, в данном алгоритме, тоже используется схема: "цикл в цикле", то временная сложность данного алгоритма: O(N^2)

В основном (внешнем цикле) количество итераций будет: N-1. Текущим, делаем по очереди каждый элемент не отсортированной части.
Во втором цикле (внутреннем цикле), количество итераций будет расти с каждым разом, поскольку, во внутреннем цикле расширяется отсортированная зона (обход желтых ячеек).

Из всех вводных данных можно сделать вывод, что алгоритм "Сортировка методом прямой вставки" сопоставим с алгоритмом: "Сортировка методом выбора", но на самом деле - это не так. 
При тонком анализе, можно сделать вывод, что алгоритм: "Сортировка методом прямой вставки" лучше, чем "Сортировка методом выбора".

*Вопрос*: Лучшая временная оценка для алгоритма: "Сортировка методом прямой вставки" и в какой ситуации она будет?
Лучшая временная оценка для алгоритма: "Сортировка методом прямой вставки" будет в ситуации, когда массив уже отсортирован, то числа не нужно никуда двигать, когда начнется внутренний цикл, он сразу и закончится.
Если все ячейки отсортированы, то внутренний цикл не будет работать вообще, а будет работать только внешний цикл, а там N-итераций, т.е. алгоритм сортировки будет выполняться N-раз.

В алгоритме: "Сортировка методом выбора", алгоритм работает независимо от ситуации в ячейках, потому что какие бы числа не были, алгоритм выполняется полностью в любом случае, поскольку, на каждом шаге необходимо искать минимум и мы не знаем отсортирован ли массив или нет и не можем узнать, есть ли минимум или его нет, пока не выполним весь алгоритм.

Алгоритм: "Сортировка методом прямой вставки" учитывает данные, которые находятся в массиве, а алгоритмом: "Сортировка методом выбора" не учитывает данные, которые находятся в массиве.

При условии, что алгоритмы: "Сортировка методом прямой вставки" и "Сортировка методом выбора" - сопоставимы по сложности, т.е. программисту написать их разницы нет, но при том, что в ряде ситуаций алгоритм: "Сортировка методом прямой вставки" работает лучше, как минимум не хуже, то выбор алгоритма очевиден.


### Сортировка методом пузырька

Пузырьковая сортировка - это сортировка обменом. 
Есть сортировка методом выбора - это группа алгоритмов, есть сортировка методом вставки - это тоже группа алгоритмов, есть сортировка методом обмена и именно к ней относится пузырьковая сортировка.

**Основная идея алгоритма**: массив условно делится на две части: отсортированную и пока не отсортированную. В цикле осуществляется проход по не отсортированной части и сравниваются соседние ячейки. Сравнивается текущий элемент и тот который ниже и если текущий больше следующего, то их нужно поменять местами.

Выполняются следующие операции:
1. В цикле выполняется проход по элементам массива от первого элемента до предпоследнего в неотсортированной части. Текущим устанавливается самый первый элемент;
2. В процессе обхода элементов выполняется сравнение текущего элемента и следующего. Если текущий элемент больше следующего, то выполняется их перестановка;
3. Текущим становится следующий элемент массива;
4. После завершения цикла неотсортированная часть массива уменьшается снизу на один элемент. Если количество оставшихся элементов больше одного, то происходит переход на шаг 1.

*Графический пример сортировки методом пузырька*:
![[Pasted image 20260410020002.png]]

Картинка не совсем верная. Алгоритм сортировки методом пузырька, обычно делается снизу -> вверх, хотя можно и наоборот.

**Результат первого прохода** по алгоритму сортировки методом пузырька - это то, что **самый маленький элемент**, оказывается **гарантировано на самом верху** из не отсортированной части.

Название алгоритма происходит из полученного результата, потому что пузырек воздуха легче воды и он поднимается на верх (маленькое число всплывает).

Если алгоритм делать сверху -> вниз, то самое большое число будет окажется в самом низу. Таким образом, самое большое число гарантировано окажется в самом низу из не отсортированной части.

**Временная сложность этого алгоритма: O(N^2) - верхняя оценка.** 

Алгоритм: "Сортировка методом пузырька" не учитывает данные, которые находятся в массиве, именно в такой трактовке, которая показана выше. 
Поскольку, в любом случае будет запущен обход ячеек, даже если массив отсортирован, сравниваем, меняем местами и т.д.

Но у алгоритма: "Сортировка методом пузырька" есть разновидность, как и у других алгоритмов сортировки.

### Сортировка перемешиванием (шейкерная)

Сортировка перемешиванием (шейкерная) (от англ. Cocktail shaker sort) - это разновидность пузырьковой сортировки.

**Основная идея алгоритма**: базируется на добавлении 3-х улучшений, относительно стандартного алгоритма пузырьковой сортировки.

Данная разновидность алгоритма пузырьковой сортировки строится на трех изменениях:
1. Движение по массиву не только снизу -> вверх, но и сверху -> вниз. Таким образом в результате чередования, при первом проходе поставим самое маленькое число на свое место, а затем поставим самое большое число на свое место;

Это необходимо того, если в массиве часть данных уже отсортирована и нужно поставить самый большой элемент на свое место.
Половина источников в интернете, дает некорректную информацию о сортировке перемешиванием (шейкерная), поскольку, если бы идея заключалась только в чередующимся проходе снизу -> вверх и сверху -> вниз, то особого выигрыша не получилось бы.

2. Использование границ просматриваемой части;
3. Были ли перестановки вообще;

Если не учитывать все три особенности в совокупности, то алгоритм: "Сортировка перемешиванием (шейкерная)" не лучше алгоритма: "Сортировка методом пузырька", так как она без улучшений будет давать такую же временную сложность: O(N^2).
Если реализовать все три улучшения, то алгоритм становиться больше по количеству операторов, но за то он оптимальнее будет работать.

В массиве устанавливаются левая (left) и правая (right) границы не отсортированной области. В начале они равны граничным индексам, но по мере работы сдвигаются навстречу друг другу. Выполняются следующие операции:

1. В цикле выполняется проход по элементам массива от индекса left до right.
2. В процессе обхода элементов сравнивается текущий элемент и его сосед справа и если необходимо, они обмениваются. Граница right устанавливается равная индексу последнего переставленного элемента.
3. В цикле выполняется проход по элементам массива от индекса right до left.
4. В процессе обхода элементов сравнивается текущий элемент и его сосед слева и если необходимо, они обмениваются. Граница left устанавливается равная индексу последнего переставленного элемента.
5. Цикл выполняется, пока left и right не станут равны.


*Графический пример сортировки перемешиванием (шейкерная)*:
![[Pasted image 20260410023808.png]]

В процессе работы используются 2 переменных: left (top) и right (bottom). 
- Переменная: "left" - левая граница не отсортированной части. Изначально она равна самому первому элементу;
- Переменная: "right" - правая граница не отсортированной части. Изначально равна самому последнему элементу;
В процессе обхода переменные: left и right будут меняться.
Если мы начинаем обход: сверху -> вниз, то самое большое число окажется на своем месте. И переменную: "right" мы делаем равной ячейке, которая последняя совершила обмен. Таким образом неотсортированная часть массива сузилась на 1 ячейку.
Далее, делаем обход: снизу -> вверх, в результате чего, самое маленькое число окажется на своем месте. И переменную: "left" мы делаем равной ячейке, которая последняя совершила обмен. Таким образом неотсортированная часть массива сузилась еще на 1 ячейку.
Далее повторяем проходы по очереди и в какой-то момент обмен не будет выполняться, если верхняя часть массива будет уже отсортирована, то последний обмен будет где-то наверху и за счет этого переменная: "left" будет смещена сразу на несколько позиций ниже и она не на 1 единицу будет увеличиваться каждый раз, а на несколько ячеек. Это позволит существенно сэкономить количество итераций.
Причем, чем больше отсортированы края массива, верхняя или нижняя часть, тем больше вероятность, что можно перепрыгнуть какое-то количество ячеек и тем самым сократить количество итераций.

В алгоритме: "Сортировка перемешиванием (шейкерная)" не только проход в разные стороны реализуется, а еще и управление границами, управление двумя переменными, которые определяют границы не отсортированной части.

### Быстрая сортировка (Quick sort) (сортировка Хоара)

Быстрая сортировка (Quick sort) (сортировка Хоара) - это оптимальная сортировка, но реализовать ее не так просто.

Быстрая сортировка (Quick sort) (сортировка Хоара) - это рекурсивный алгоритм.

В алгоритме выполняются следующие операции:
1. Выбрать из массива опорный элемент с которым на следующем шаге будет происходить сравнение;
2. Сравнить все элементы массива с опорным и переставить их так, чтобы элементы меньше опорного были выше него, а остальные элементы были ниже него;
3. Для части массива выше опорного и части массива ниже опорного рекурсивно повторить действия с п.1;

*Графический пример быстрой сортировки (Quick sort)*:
![[Pasted image 20260410191727.png]]

**Основная идея алгоритма**: Есть массив. Есть границы массива, с которым работаем. В одной части массива, устанавливается опорный элемент (на рисунке выделен синей стрелочкой). Он может быть любым, но правда есть определенные принципы, как более оптимально выбрать опорный элемент. По вопросу корректного выбора опорного элемента, существуют целые исследования.
Далее проходимся по всем ячейкам массива и переставляем элементы в таком порядке, чтобы те элементы, которые меньше опорного,  оказались выше него, а те которые больше - ниже (перекидываем элементы местами на 2 кучки).
В результате такого прохода, массив делится на 2 части:
- Первая часть - это элементы у которых значение меньше опорного;
- Вторая часть - это элементы у которых значение больше или равно опорному;
Далее функция вызывается рекурсивно для каждой части.

**Стратегия от предложенная преподавателем по выбору опорного элемента**: необходимо выбирать опорным элементом тот, который первым стоит не на своем месте.

*Вопрос*: Когда рекурсия заканчивается? 
- Рекурсия заканчивается тогда, когда остался один элемент или 2 элемента, но если элемента 2, то могут быть ситуации, когда их придется поменять местами.
- Если рекурсия вызвана для определенной части массива, мы по ней прошлись и выявили, что она больше не нужна, то ее не нужно вызывать;

Алгоритм выбора опорного элемента нужен для того, чтобы узнать отсортирована ли необходимая часть в массиве или нет.
Когда мы выбираем опорный элемент, согласно стратегии предложенной преподавателем, мы берем опорным элементом ячейку, которая стоит не на своем месте, для этого мы вынуждены сделать проход по ячейкам, чтобы понять, стоят ли все ячейки на своих местах и если все ячейки стоят на своих местах, то ничего не нужно делать: выбирать опорный элемент, перекидывать ничего никуда не нужно,  рекурсию не нужно вызывать функцией.

**Временная сложность этого алгоритма: O(n logN)**.

Если посмотреть в документации к C# в классе: "Array", то там используется именно быстрая сортировка (Quick sort), но на самом деле, там различные алгоритмы, которые зависят от разного размера массива, но основой является алгоритм быстрой сортировки (Quick sort).








# Лекция 3: Обработка ошибок

## Типы ошибок

1. **Синтаксические**;
Ошибки данного типа допускаются при наборе текста программы, в современных редакторах кода с ними проще всего, так как сама среда их выделяет и подсказывает, как их исправить.
В любом случае, программу не получится запустить на исполнение, не получится откомпилировать, пока все синтаксические ошибки не будут устранены.

2. **Логические**;
С данным типом ошибок, значительно сложнее, поскольку, ошибки данного типа не фиксируются компилятором и он не может подсказать, где именно она находится. Часто, ошибка не из-за одной строки или знака, а является следствием целой серии неверных действий.
Логические ошибки невозможно определить при компиляции, а можно определить, когда программа работает, да и то не всегда, так как при определенных стечениях обстоятельств, при определенных значениях переменных, ошибка может и не проявиться.
Как находить такие ошибки, это отдельная тема для обсуждения, есть целая IT профессия, которая этим и занимается.
Есть большая теоретическая база, какие вообще бывают методы обнаружения ошибок. Также в учебном плане есть отдельная дисциплина по тестированию программного обеспечения.

3. **Ошибки времени выполнения (run-time error), исключительные ситуации (exception)**.
Данный тип ошибок возникает не потому что некорректный синтаксис или сам по себе не верный алгоритм, а обычно главный источник ошибки - это внешняя среда.
При взаимодействии программы с внешней средой, которая может подавать некорректные данные, на что очень часто невозможно повлиять и если ничего не сделать, то программа аварийно завершит работу. Но бывают ситуации, когда это нормально, когда программа служебного или системного характера.
Пример ситуации, когда программа аварийно завершает работу и это приемлемая ситуация: для личного пользования, была написана программа, которая автоматизирует действия с файлами. Программа обходит все папки и подпапки, которые находятся по указанному пути, находит дубликаты и перемещает дубликаты в другое место. При указании пути, папка оказывается защищена, программа закрывается, аварийно завершает работу, но такой сценарий не является не приемлемым, так как понятна суть ошибки, легко можно исправить или указать другой путь.
Стоит отметить, что приведенный пример - это исключение из правил и для большинства ситуаций такое поведение не нормальное, особенно, когда программа пишется для простых пользователей, которые не знают технические детали или если программа работает с данными, то при аварийном завершении программы, все данные потеряются, все, что находится в оперативной памяти - это все будет уничтожено. Чаще всего, на исключительные ситуации нужно реагировать.
Самый частый пример исключительных ситуаций, которые нам встречаются в тех задачах, которые мы решаем, если пользователь вводит некорректные данные, например, когда пользователь вводит вместо числа, какие-либо знаки, то функция: Parse() при конвертировании строки в число - выдаст ошибку. В указанном примере, внешняя среда - это пользователь.
Другой пример, файловая система: сохранение файла с именем, которое уже есть, отсутствие доступа, повторное открытие файла, который открыт в другом процессе и т.д.

**Внешняя среда** - это понятие очень широкое, это и какие-либо устройства (связываемся с устройством, ожидаем обратную связь, а устройство некорректно себя ведет), другие программы (обращаемся к серверу, он либо не отвечает, либо дает некорректный ответ), файловая система, пользователь.

Игнорировать ошибки, чаще всего - это не правильно и нужно научиться их обрабатывать.

**Структурированная обработка исключений** – методика, предназначения для обработки исключений, которые могут возникать в процессе выполнения программного кода.


## Подходы к обработке ошибок

Обработка ошибок - это не только оператор: **Try**. есть разные варианты. В некоторых ситуациях оптимален один вариант, в других ситуациях оптимален другой. Необходимо знать и понимать в каких случаях оптимален тот или иной вариант.

Выделяют 3 подхода к обработке ошибок:
1. **С использованием операторов условия (If)**;
Идея использования проста. До того, как сделать операцию, нужно с помощью оператора: "if" или системы операторов: "if" проверить, а сможем ли мы это сделать.
Например, прежде чем записывать что-то в файл или папку, проверить, если ли доступ к файлу или папке с помощью функции.
Данный способ не всегда удобен, так как таких проверок иногда приходится писать очень много.

Второй и третий вариант похожи. Основная их идея в том, что до операции ничего не проверяется выполняется операция и если при выполнении возникает ошибка, то необходимо на нее реагировать. Проверка осуществляется не **до**, а **постфактум**.

2. **Не структурированный: использование локальных констант, констант API-интерфейса, COM-интерфейсов**;
Это в основном библиотеки старого типа, если же говорить о платформе: ".Net", то необходимо использовать третий вариант.

3. **Структурированная обработка исключений – structured exception handling**. 
	 - **Унифицированный подход**. Принципы обработки ошибок одинаковы, независимо от ситуации.;
	 - **Объектно-ориентированный подход**. Суть в том, что когда ошибка случается, то создается объект на основе некоторого класса и мы с ним работаем и можем использовать его свойства и методы;


## Сущности структурированной обработки исключений в C\#

1. Тип класса ошибки;

Есть понятие, класс ошибки. Основная идея в следующем: 
Все операторы, которые могут потенциально произвести ошибку, мы заключаем в блок оператора: "Try" (от англ. попробовать, попытаться) и это значит, что если на любой строчке этого блока случиться ошибка, то выполнение кода перейдет к плану: "Б", вместо запланированного порядка выполнения операторов, будут выполняться другие действия, если в этом блоке возникнет ошибка.

При возникновении ошибки, система автоматически создает объект, с которым можно работать или не работать. Использование данного объекта позволит дать информацию о типе ошибки, что бывает важно и дать дополнительную информацию.

Бывают сложные случаи, когда ошибка возникла не в наших операторах, а когда мы вызываем функцию, она вызывает еще функцию, которая вызывает еще функцию и в данной функции возникает ошибка, а сама функция находится не в нашем проекте и написана не нами и физически находится в другой библиотеке.
Поэтому, бывают ситуации, когда необходимо определить, где возникла ошибка и как раз это можно сделать с помощью объекта, который создается при использовании блока: "Try".
Объект создается автоматически, на базе класса той ошибки, которая случилась. Система классов уже есть в Visual Studio в библиотеках ".Net", все они используют наследование от базового класса: "Exception", который находится в пространстве имен: "System", но сами классы, часто другие, но они производные от "Exception".
Также есть возможность создать свой класс ошибки.

2. Член класса – метод (функция), который способен 
генерировать (throw) в вызывающем коде экземпляр 
класса исключения при некоторых обстоятельствах

Нужно понимать в каком методе случилась ошибка, тот блок кода, который потенциально ошибку сформировать.
Ошибка может произойти в результате каких-то действий или обратной связи от внешней среды. 
Но бывают ситуации, когда мы можем сами сгенерировать ошибку и делается это с помощью оператора: **"Throw"** (от англ. кидать/бросать).

Таким образом, программный код можно разделить на 2 категории:
- Программный код, который ошибку сформировал;
- Программный код, который обрабатывает ошибку;

Обрабатывать ошибку можно не в той функции, которая ее сформировала, чаще всего обработка ошибки происходит не в функции, которая сформировала ошибку.

3. Блок кода на вызывающей стороне, ответственный 
за обращение к члену (функции), в котором может 
произойти исключение
4. Блок кода на вызывающей стороне, который будет 
обрабатывать (перехватывать – catch) исключение в 
случае его возникновения

Ключевые слова (Keywords): try, catch, finally, throw

## Пример использования Try/Сatch

- Блок: "try" содержит защищенный код;
- Блок: "catch" может быть без аргументов;

```C#
// Пример использования Try
class Program
{

	static void Main(string[] args)
	{
	
		int x, y, res;
		x = 10; y = 0; 
		res = 0; // инициализируем переменную, чтобы при завершении программы, в переменной было значение.
		
		try
		{
			res = x / y;
		}
		catch
		{
			Console.Writeline("Error");
			// res = 1 - можно поставить знаечние по умолчанию
			return; // аварийно завершит работу функции
		}
		
		Console.Writeline(res);
	
	}
}
```

Как именно будут использоваться операторы: "try", "catch" - это решать программисту, будет ли аварийное завершение программы или подстановка значений по умолчанию или как-либо еще.


## Пример использования Try/Сatch с Exception

Если необходимо получить информацию об ошибке, то можно воспользоваться аргументом общего типа Exception в блоке: "Сatch".
Если в блоке: "Сatch" объявлена переменная типа (класса): "Exception" (*пример: catch (Exception ex)*), то в данный блок программа будет заходить, если ошибка будет именно данного типа, если ошибка другого типа, то в блок: "Сatch" с переменной типа (класса): "Exception" программа заходить не будет.
Однако, класс: "Exception" - это общий родительский класс и при таком раскладе в данный блок программа будет заходить всегда.
В блоке: "Сatch" можно объявить переменную специализированного типа (класса), тогда система будет смотреть по совпадению типа ошибки.

Блоков: "Сatch" может быть несколько, данный блок очень похож на блок: "Case" в операторе: "Swich".

```C#
// Пример использования Try/Сatch с Exception
class Program
{

	static void Main(string[] args)
	{
	
		int x, y, res;
		x = 10; y = 0; 
		
		try
		{
			res = x / y;
			Console.Writeline(res);
		}
		catch (DivideByZeroException ex) // ex - это имя переменной, оно может быть любое; 
		{
			Console.Writeline(ex.GetType());
			Console.Writeline(ex.ToString());
			Console.Writeline(ex.Message);
			res = 1;
			
		}
		
	}
}
```

В любом объекте, можно использовать метод: "ToString()", который сопоставляет строку с данным объектом. При его использовании, он может выдать полную информацию об ошибке, вплоть до той строки, где она произошла. 
Данный вывод информации бывает полезен, чтобы узнать класс, в котором находится ошибка.

В режиме отладки можно использовать метод: "GetType()" для того, чтобы уточнить тип ошибки и получить имя класса. Его удобнее использовать, чем метод:  "ToString()".

## Пример использования Try/Сatch с FormatException

Порядок действий при обработке ошибок разными сценариями, обработка разных видов ошибок различными действиями:
1. Нужно определить в каких операторах ошибка возможна;
2. Написать блок: "Try";
3. Дальше нужно написать блок: "Сatch" (обобщенный);
4. Дальше нужно написать блок: "Сatch" (частный) и поставить его выше общего;

Порядок расположения блоков: "Сatch" играет роль, необходимо их располагать от частного к более общему.
Работает также, как и в операторе: "Swith", в каком порядке блоки: "Case" расположили в таком порядке они и проверяются. Таким образом, если первым расположить блок: "Сatch" с "Exception", то до блока: "Сatch" с "FormatException" очередь никогда не дойдет.


```C#
// Пример использования Try/Сatch с Exception с разными видами ошибок
class Program
{

	static void Main(string[] args)
	{
	
		short x;
		string s;
		
		s = Console.ReadLine();
		
		
		Console.Writeline(x * 10); 
		
		try
		{
			x = short.Parse(s);
		}
		catch (FormatException ex)
		{
			Console.Writeline("Вы ввели не число!");
		}
		catch (OverflowException ex)
		{
			Console.Writeline("Слишком большое число!");
		}
		catch (Exception ex)
		{
			Console.Writeline(ex.GetType());
			Console.Writeline(ex.Message);		
		}
		
	}
}
```


## Пример использования Try/Сatch c уточнением ошибки (when)

При использовании формата переменной: "ushort", числа: 900000 и -90 вызовут один и тот же тип ошибки: "OverflowException".

Если необходимо различать ошибки, то для уточнения ошибки можно использовать необходимо использовать служебное слово: "when", после объявления переменной в блоке: "Сatch".

Использовать служебное слово:  "when" не получится на результирующей переменной, так как она не посчитается при возникновении ошибки, поэтому, нужно придумывать иные методы решения данной проблемы.
 
```C#
// Пример использования Try/Сatch с служебным словом when
class Program
{

	static void Main(string[] args)
	{
	
		short x;
		string s;
		
		s = Console.ReadLine();
		
		
		Console.Writeline(x * 10); 
		
		try
		{
			x = short.Parse(s);
		}
		catch (FormatException ex)
		{
			Console.Writeline("Вы ввели не число!");
		}
		catch (OverflowException ex) when (s[0] == '-') // проверяем первый символ введенного значения
		{
			Console.Writeline("Введено отрицательное число!");
		}
		catch (OverflowException ex)
		{
			Console.Writeline("Слишком большое число!");
		}
		catch (Exception ex)
		{
			Console.Writeline(ex.GetType());
			Console.Writeline(ex.Message);		
		}
		
	}
}
```

Приведенный пример - это простой учебный пример, не нужно искать в данном примере глубокий смысл. Основная идея заключается в том, чтобы показать, как это работает, изучить синтаксис и понимать общий смысл.


## Рассуждения про использование блоков: Try/Сatch 

Самое сложное в этой теме - это грамотная организация самого процесса объявления ошибки.

Какие операторы необходимо заключать в блок: "Try".
В блок: "Try" не нужно заключать всю функцию из 20 строк, так как возможны ошибки разных типов, а следовательно блоков: "Catch" будет много и это неудобно читать. Самое главное, что в  разных частях функции ошибки могут быть одинаковые, но поведение может отличаться.
К примеру, пользователи вводят буквы вместо числа, в одном месте нужно будет попросить пользователя ввести информацию заново, а в другом месте, такую же ошибку необходимо по другому прорабатывать.
К примеру, читается информация из файла, зачем пользователю сообщать, что программа не смогла преобразовать информацию, так как пользователь не сможет все равно ничего сделать, проще подставить значение по умолчанию.
Разные операторы для одной и той же ошибки и это можно сделать с помощью разных блоков: "Try".

Правильнее всего в коде определять где ошибка потенциально может быть и какого она вида, писать именно для этой строки блок: "Try" и писать свои блоки: "Catch". Если ошибка одного типа, то блок: "Catch" один, а если ошибка разного вида, то тогда писать разные блоки: "Catch" и по разному реагировать на ошибки.
Те операторы, где ошибка не может случиться, не нужно включать их в блок: "Try".

Есть еще вариант, когда известно, что ошибка может случиться, но она  сознательно не обрабатывается, так как когда случается ошибка, она передается вызывающей стороне.

Разбор случая: 
В главной функции: "main", просим пользователя ввести имя файла. Пользователь вводит. Далее вызывается функция и передается ей имя файла. Функция должна найти файл и его обработать по определенному алгоритму. Но в процессе обработки файла, возникает ошибка.
Вопрос: В какой функции необходимо обрабатывать ошибку? Ошибку необходимо обрабатывать в функции: "main", именно в ней писать блок: "Try", так как она взаимодействует с пользователем, а в функции, которая обрабатывает файл, писать блок: "Try" не имеет смысла, так как она не взаимодействует с пользователем.
Странная ситуация, когда в функции общего назначения, написан блок: "Try", поскольку, функция не взаимодействует с пользователем, а в ней что-то обрабатывается и она что-то выдает в консоль. 

## ## Пример использования Try/Сatch c блоком Finally

Блок: "Finally" немного странный. Блок не является обязательным и пишется в самом конце, после всех блоков: "Сatch".

Блок: "Finally" выполняется в любом случае – и если ошибки не случилось (программа корректно сработала), выполнили команды в блоке: "Try" и потом в блоке: "Finally", либо когда ошибка произошла, выполнили команды в определенном блоке: "Сatch", а затем в блоке: "Finally".

Если блок: "Finally" - отсутствует, то будет тоже самое, просто выполнение кода продолжится ниже.

Обычно в блок: "Finally" помещают операторы для освобождения ресурсов, финализации операций, задания значений по умолчанию и т.п. - это объекты, которые необходимо уничтожить, перед завершением работы программы, если есть открытые файлы, которые нужно закрыть, если есть открытые сессии, то их тоже нужно закрыть, открытые БД, открытые подключения к серверу, т.е. любые действия, которые необходимо сделать, перед тем, как программа закроется.

Ошибка может случиться на моменте подключения к ресурсу, т.е. при подключении к базе данных, возникла ошибка, программа перешла в блок: "Сatch", потом программа пойдет в блок: "Finally" и попытается сделать отключение от базы данных, но она не подключена и может опять возникнуть ошибка.

В блоках: "Сatch" или "Finally" могут тоже возникать ошибки, поэтому в блоках: "Сatch" или "Finally" нужно писать максимально надежные операторы, которые точно не произведут ошибки или проверять оператором условия: "if"

Блок: "Finally" не самостоятельный оператор, а часть оператора: "Try".

```C#
// Пример использования Try/Сatch с finally
class Program
{

	static void Main(string[] args)
	{
	
		short x;
		string s;
		
		s = Console.ReadLine();
		
		
		Console.Writeline(x * 10); 
		
		try
		{
			x = short.Parse(s);
		}
		catch (FormatException ex)
		{
			Console.Writeline("Вы ввели не число!");
		}
		catch (OverflowException ex) when (s[0] == '-') // проверяем первый символ введенного значения
		{
			Console.Writeline("Введено отрицательное число!");
		}
		catch (OverflowException ex)
		{
			Console.Writeline("Слишком большое число!");
		}
		catch (Exception ex)
		{
			Console.Writeline(ex.GetType());
			Console.Writeline(ex.Message);		
		}
		finally
		{
			Console.Writeline("Finally");
		}
		
	}
}
```

## Распространение (проброс) ошибки

Наличие ошибки, вначале определяется программистом по тексту программы, может потенциально ошибка возникнуть в той или иной строке или не может;
Далее необходимо определить, необходимо ли обрабатывать ошибку по месту или не нужно. От этого будет зависеть, включен ли обработчик ошибок или нет (есть ли блок: "Try"). Если обработчика нет, то ошибка распространяется, пробрасывается по стеку вызовов функции на верх. Ошибка возникла в функции: "a", она там не обрабатывается, значит все остальные операторы в функции не выполняются, управление передается вызывающей стороне, к примеру, функции: "b", если и там нет блока: "Try", то управление передается еще выше по стеку и т.д. до самой первой функции (обычно, это функция: "main") и если там нет блока: "Try", то программа аварийно завершает работу. 

*Краткое описание алгоритма распространения ошибки*:
1. Включен обработчик или нет;
2. Распространение ошибки вызывающей стороне;
3. Возникновение ошибки в обработчике ошибки;


## Алгоритм обработки ошибок в коде

1. Определить блоки кода, в которых потенциально возможны ошибки;
2. Сгруппировать по вариантам обработки;
3. Определить способ обработки:
	- По месту;
	- В вызывающей стороне;
4. При обработке по месту написать оператор try и возможные варианты (сценарии) ошибок (catch);
5. Проверить возможность возникновения ошибки в обработчике ошибок;

!!!Важно!!! Необходимо определиться, кто будет ответственный за ошибку, т.е. определить, кто будет нести ответственность за обработку, наша функция, полномочна ли она что-то с этим сделать или все же вызывающая сторона.

Необходимо также использовать различные приемы для упрощения обработки ошибок.
К примеру, можно написать свою функцию, записать в нее необходимый код и вызывать ее, когда это необходимо.

## Генерация ошибки оператор Throw

Оператор: "Throw", позволяет ошибку генерировать.
Например, это может быть использовано, когда пишется функция или метод, которым передаются какие-либо данные через аргументы и данные некорректны и метод должен как-то реагировать.

Ранее мы писали функции и возвращали признак успешности их выполнения, такой вариант тоже годиться.
Другой вариант не возвращать никакой bool, а делать ошибку и пусть вызывающая сторона обрабатывает

- Оператор throw ex;  или  throw new Exception;
- Для генерации ошибки должны быть веские основания;
- Создание и использование своего класса для особого типа исключений. Рекомендуется наследовать от System.ApplicationExeption

```C#
// Пример использования оператора Throw
public struct Human
{
	public string Name {get; set;}
	public uint Age
	{
		get;
		set
		{
			if (value > 200) throw new OverflowException();
		}
	}
}


class Program
{
	static void Main(string[] args)
	{
		Human h = new Human();
		
		try
		{
			h.Age = 300;
		}
		
	}
}
```
